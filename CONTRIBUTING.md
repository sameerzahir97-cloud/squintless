# Contributing to Squintless

Thanks for helping make terminals easier on the eyes. Squintless is a small,
MIT-licensed project — a cross-platform, eye-strain–optimized terminal + Claude Code
setup. Bug reports, new terminals, new package managers and new palettes are all
welcome.

For anything large (a new variant, a new terminal target, restructuring the
installer), please **open an issue first** so we can agree on the shape before you
write much code.

## Get the repo

```bash
git clone https://github.com/sameer-zahir/squintless.git
cd squintless
```

You don't need the installer to develop. The validators and the generator are
**stdlib-only Python 3** — no `pip install`.

## Repo layout

| Path | What lives here |
| --- | --- |
| `install.ps1` | Windows installer (PowerShell 7+). The live `irm …/sq \| iex` one-liner. |
| `install.sh` | macOS / Linux installer (bash 3.2-safe). The live `curl …/sh \| bash` one-liner. |
| `palettes/` | **Single source of truth** — `tokyo-night-moon.json` (dark) + `gruvbox-light.json` (light). |
| `tools/` | `gen.py` (renders schemes from palettes) and `validate.py` (JSON + scheme + version-sync checks). |
| `config/` | Hand-maintained configs the installers place (oh-my-posh, delta, PowerShell profile snippets, Windows Terminal scheme files). |
| `config/generated/` | **Generated — do not hand-edit.** Per-terminal schemes + the WCAG table, written by `tools/gen.py`. |
| `module/` | The `Squintless` PowerShell Gallery module (`Invoke-Squintless`). |
| `claude-plugin/` | The Claude Code plugin + the `/squintless-setup` skill. |
| `worker/` | The Cloudflare Worker behind `sameerzahir.com/sq` and `/sh` (302 redirects to `main`). |
| `bucket/` | The Scoop manifest (`squintless.json`). |
| `tests/` | `Squintless.Tests.ps1` (Pester) + `install.bats` (bats). |
| `.github/workflows/` | CI: `lint-and-validate`, `windows`, `unix`, `release`. |

## Colors: edit palettes, never the generated files

Every per-terminal color scheme is **generated from `palettes/*.json`**. There is
exactly one place to change a color.

1. Edit `palettes/gruvbox-light.json` (light) and/or `palettes/tokyo-night-moon.json`
   (dark).
2. Regenerate:
   ```bash
   python tools/gen.py
   ```
   This rewrites everything under `config/generated/**` (kitty, Alacritty, WezTerm,
   Ghostty, iTerm2, Xresources, the Windows Terminal fragments), refreshes the two
   `config/windows-terminal.scheme*.json` files the live installer reads, and
   recomputes `config/generated/wcag-contrast.md`.
3. Commit the palette change **and** the regenerated output together.

Never hand-edit anything in `config/generated/` — each file is stamped
"generated by tools/gen.py", and CI fails if it's out of date (`python tools/gen.py
--check`). If you're tempted to tweak a generated file directly, change the palette
instead.

## Run the checks locally

Before opening a PR, run what CI runs:

```bash
# Everywhere (stdlib Python 3):
python tools/validate.py        # JSON well-formedness, scheme completeness, version sync
python tools/gen.py --check     # fails if any generated file is stale
```

```powershell
# Windows: Pester unit tests for install.ps1 (loads only its functions; touches nothing real)
Invoke-Pester -Path tests
```

```bash
# macOS / Linux: lint + sandboxed installer tests
shellcheck --severity=style install.sh
bats tests/install.bats         # each test runs in a throwaway $HOME
```

`tests/install.bats` is safe to run on your own machine — every test redirects `$HOME`
and git's global config to a temp dir, so it never touches your real setup. Pester
loads `install.ps1` via the AST and exercises only its functions, so it also makes no
system changes.

## Keep the version strings in sync

A release is one version, everywhere. `tools/validate.py` enforces that these match,
and a tag push must equal `VERSION`. If you bump the version, change **all five**:

- `VERSION`
- `install.ps1` → `$SquintlessVersion`
- `install.sh` → `SQUINTLESS_VERSION`
- `module/Squintless/Squintless.psd1` → `ModuleVersion`
- `claude-plugin/.claude-plugin/plugin.json` → `version`

(Most PRs shouldn't bump the version at all — releases are cut separately. Only touch
these if your change ships a new version, and say so in the PR.)

## Opening a pull request

- Keep changes focused; describe what changed and why.
- If you changed colors, confirm you regenerated with `python tools/gen.py` (not by
  hand) and committed the result.
- Make sure `python tools/validate.py` and `python tools/gen.py --check` pass.
- Test the installer on the OS you touched, if you touched it.

The PR template will remind you of all of this.
